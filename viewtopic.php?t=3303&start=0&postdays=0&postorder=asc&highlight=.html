ich hab mein aktuelles Projekt mal auf Performance unter Last getestet: 30 Requests pro Sekunde und das "nur" 5 min
Ergebnis: das Script an sich steckt das locker weg, nur MySQL kratzt ab -.-

Lösungen, die ich kenne:
MySQL-Cluster
kann hier leider nicht zum Einsatz kommen, da die Daten zu groß für den RAM eines Servers sind. Auf die Festplatte kann ich die Daten mit dem Cluster leider auch nicht packen, das ich die Indizierung zwingend brauche.

Master-Slave-System
Das ist mein bisheriger Favorit: Ein (leistungsfähiger) Server für Schreibzugriffe und mehrere Slaves nur zum Lesen. => Bessere Lastenverteilung.

Die Datenbank wird kapp 1000 Tabellen umfassen. Pro Request wird es sessionbedingt mindestens 2 Schreibzugriffe und auch mindestens 2 (wenn nicht mehr ) Lesezugriffe geben.
die Datenbank ist im Ausgangszustand knapp 6 GB groß und wächst mit der Zeit natürlich mit.
Indizes, Primary key und Unique Keys sind schon drin :

Kennt jemand noch weitere Lösungen?
Nein, das ist nicht sinnlos: Das letzte Projekt, was aus dieser "Sparte" kam, ist mit ~30 Zugriffen/Sekunde abgekackt -gleich am Tag der Veröffentlichung. Genau das soll meinem Projekt nicht passieren

Eine einfache Master-Slave-System Lösung kann hier durchaus, wie du schon sagst, ausreichend sein.

Es ist übrigens auch oftmals hilfreich MySQL funktionstechnisch auch auszureitzen. Du sagst du hast bis zu 2 Schreibzugriffe pro Request. Ich empfehle hier diese beim Zugriff in eine Memory Tabelle durchzuführen die über keine Indexe verfügt, und sie Zeitversetzt gebündelt in die richtige DB zu schreiben, sofern das geht (z.B. für Logs).

Wenn es nicht geht weil du die Daten direkt auch brauchst kannst du auch bei der Memory Lösung+Indexen bleiben, oder z.B. Memcache nutzen.

Der Beste Weg für soetwas ist aber natürlich immer die Optimierung der Datenbankabfragen und das Nutzen eines caches.
Versuche auch immer nur die Daten zu laden die du wirklich brauchst, nutze keine Select *, auch sortiere nur dort wo es nötig ist. Auch Group By ist kein guter Freund, LIMIT hingegen schon
